# 当前已知限制

当前版本的快速导出主线已经覆盖学校规范驱动的结构与版式基线。Workbench v1 已接入项目空间、版本和建议队列骨架，但仍有以下边界：

## 自动保证

- 正式封面、目录字段、页眉页脚、页码、分节、固定章节顺序
- 关键字体、字号、段落、悬挂缩进与留白策略
- `.docx` 与文本输入的统一结构化解析
- Workbench 项目、文件、版本、导出记录的本地持久化
- 未确认 Proposal 不会改变当前导出版本
- Provider 配置不向前端返回密钥，自定义 base URL 有 SSRF 防护
- 项目级隐私模式、远程 Provider 授权提示和访问码保护

## 仍需人工复核

- 原始 `.docx` 中的表格
- 图片、绘图对象、文本框、形状
- 脚注、尾注与复杂域代码
- 个别参考文献条目的格式细节
- Word 打开后目录字段更新结果
- PDF 本地粗解析结果
- 图片/OCR 占位结果
- 老师批注的复杂锚点匹配
- Workbench 中复杂正文 diff 的语义正确性

## 不会做的事

- 不会为了让文档“看起来完整”而自动补内容
- 不会伪造参考文献元数据
- 不会承诺任意 Word 文档的高保真无损修复
- 不会预测查重率
- 不会把未确认联网来源作为合规依据
- 不会把实验性 Story2Paper 正文直接写入默认导出版本

## 尚未做实

- Celery / Redis 长任务队列仍是部署骨架，当前解析任务同步执行
- MinIO/S3 SDK 尚未替换本地文件系统 adapter
- PDF 转换目前降级保留 `.docx` 产物
- 真实 LLM Provider 调用尚未接入 Agent Runtime
- Access code 是单机/私有部署保护，不是多用户权限系统
- Alembic 迁移脚本尚未建立，当前用 SQLAlchemy metadata 初始化和幂等 schema bootstrap

## 结论

当前导出结果可以作为本科论文送审稿基线，但复杂元素仍应结合人工复核完成最终确认。
