# Product Scope v0.2

## 项目定位

`scnu-thesis-portal` 首轮上线目标，不是“在线完整替代论文排版流程”，而是：

让用户能在网页里上传 `.docx` 或粘贴正文，完成结构识别、字段补全、章节预览与必要修正，并导出符合当前主线模板要求的规范化产物。

## 为什么首轮主产物是 `.tex` 工程 zip

当前代码链路虽然已经能在本地生成 PDF，但它建立在以下前提上：

- 本地 XeLaTeX 环境
- 本地文件系统
- 本地线程任务与 job 目录
- 本地调试日志落盘

如果不重构就直接搬到 Vercel，会带来这些问题：

- 线上函数时长与资源不确定
- TeX 依赖与模板包体积增加维护成本
- 本地 job 目录和 polling 模式不适合 serverless
- 编译失败排查复杂，线上稳定性难以承诺

因此，首轮生产环境的默认主产物应当是：

- 规范化 `.tex` 工程 zip

而不是：

- 把“线上 PDF 编译成功”当作首轮硬承诺

只有在后续完成以下条件后，才考虑恢复线上 PDF：

- 查清并验证 Vercel 最新运行约束
- 完成真实预览环境实测
- 证明 PDF 编译在时长、依赖、成功率、维护成本上可接受

若上述条件不能满足，则生产环境默认隐藏 PDF 按钮，仅保留本地开发模式的 PDF 能力，并在界面中明确说明。

## 首轮上线要解决的问题

- 学生手里只有 `.docx` 或原始正文，不知道如何进入 LaTeX 模板
- 不同章节与字段在现有模板中的落点不透明
- 用户需要一个网页内可预览、可修正的中间层，而不是一次性黑盒导出
- 当前项目已有解析和生成核心，但还没有对普通用户可用的产品流程

## 首轮上线范围

### 输入

- 上传 `.docx`
- 粘贴正文文本

### 系统处理

- 识别摘要、Abstract、正文章节、参考文献、致谢、附录
- 输出 warnings / parse errors
- 让用户在网页内补全与修正字段

### 输出

- 导出规范化 `.tex` 工程 zip

### 页面体验

- 首页写清支持范围与不支持范围
- 主流程改为同步向导
- 支持 review 与修正，而不是直接跳结果页

## 首轮明确不做

- 不承诺线上 PDF
- 不承诺保留原 Word 样式
- 不承诺支持 `.doc`
- 不承诺复杂表格、图片、脚注、特殊排版完整恢复
- 不承诺自动修复全部参考文献格式
- 不承诺学校官方规范已核验完成
- 不承诺一键覆盖本科/硕士/博士全模板

## 对外表达原则

- 明确说明当前主线模板为 `latex-scnu-web`
- 明确说明当前不是学校官方认证工具
- 明确说明复杂 `.docx` 可能需要用户手动修正
- 明确说明最终论文仍需用户自行校对
