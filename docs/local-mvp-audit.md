# 本地 MVP 审查

## 1. 当前仓库实际已有内容

当前仓库本质上仍是资料型仓库，主要包含：

- 三套上游 LaTeX 模板：
  - `templates/upstream/latex-scnu`
  - `templates/upstream/scnuthesis-scnu`
  - `templates/upstream/scnuthesis-zsben2`
- 项目定位与 PRD 文档：
  - `README.md`
  - `docs/PRD-v0.1.md`
  - `docs/inventory.md`
- 一份研究输入材料：
  - `docs/research/SCNU-Thesis-Portal-可行性与实施方案研究报告.docx`

在开始本轮开发前，仓库中不存在网站代码、后端服务、转换链路、工作模板区、输出管理目录或测试夹具。

## 2. 当前缺失哪些网站与转换能力

当前仓库缺失的关键能力包括：

- 面向用户的本地网页入口
- `.docx` 上传、解析和结构识别能力
- 结构化文本输入能力
- 内容到论文模板的映射能力
- 工作模板复制与生成工程能力
- 本地 LaTeX 编译与结果导出能力
- 错误处理、日志和任务状态查询能力
- 本地可复现的测试与验收夹具

## 3. 三套模板中哪一套最适合做第一条可跑通主线

本轮主线模板选择为：`templates/upstream/latex-scnu`

## 4. 为什么这样选

选择 `latex-scnu` 的原因是：

- 面向本科论文场景，更贴近普通学生用户的第一需求。
- `main.tex` 入口直接，目录层次清晰。
- `abstract/ body/ reference/ thanks/ appendix` 的拆分非常适合“结构化映射”方案。
- 生成链路可以集中在少量 `.tex` 文件覆写，不需要一上来处理更复杂的学位论文选项。

另外两套模板本轮不作为主线：

- `scnuthesis-scnu`：更成熟，但偏硕博主模板，字段和选项更复杂。
- `scnuthesis-zsben2`：技术栈更新，但首次编译门槛更高，当前本地环境下依赖更重。

## 5. 这一轮明确不做什么

本地 MVP 明确不做以下内容：

- 不做任意 Word 原始样式的无损保留
- 不做“上传后自动修成学校格式”的万能格式修复
- 不做多模板并行支持
- 不做 `.doc` 支持
- 不做在线编辑器
- 不做参考文献 `.bib` 智能生成与文内引用自动修复
- 不承诺当前输出已经过学校官方规范逐项核验

## 6. 本地 MVP 的真实功能边界

本轮的真实边界是：

- 输入方式仅支持：
  - 上传 `.docx`
  - 结构化表单输入
- 系统把输入内容抽取为结构化章节和字段
- 系统把这些字段映射到本科论文工作模板
- 系统在本地生成 `.tex` 工作工程
- 系统调用本地 XeLaTeX 编译 PDF
- 系统输出 PDF 和 `.tex` 工程 zip

换句话说，本轮是“结构化内容映射到模板”的 MVP，不是“现成论文一键完美修复”的产品。

## 7. 风险点与规避方案

### 7.1 TeX 环境不完整

当前机器已具备：

- `python3`
- `node`
- `npm`
- `uv`
- `xelatex`
- `bibtex`
- `soffice`

但不具备：

- `latexmk`

并且当前 TeX Live basic 缺少：

- `titlesec.sty`
- `titletoc.sty`

这意味着本地 MVP 必须把“TeX 依赖缺失”作为一等错误处理场景，而不是假设环境天然完整。

规避方案：

- 后端在任务执行前做 TeX 预检
- `/api/health` 返回依赖状态
- 页面明确展示缺失依赖，而不是让编译阶段直接报 500

### 7.2 `.docx` 结构不稳定

不同 Word 文档的标题样式、章节命名和摘要写法不统一。

规避方案：

- 仅支持 `.docx`
- 标题识别采用多级策略：
  1. Heading 样式
  2. Markdown 标题
  3. 中文章节关键词
- 识别失败时退化为“正文”单章节，而不是直接崩溃

### 7.3 参考文献自动化风险过高

直接把任意参考文献文本转成 `.bib` 或可编译引用链，第一版风险太高。

规避方案：

- 第一版只接受逐行文本参考文献
- 后端生成手工 bibliography block
- 不承诺自动处理 citation 交叉引用

### 7.4 上游模板污染风险

如果直接在 `templates/upstream/` 上开发，会把上游材料改乱。

规避方案：

- 复制出 `templates/working/latex-scnu-web`
- 每次任务再从 working 模板复制到 `tmp/jobs/{job_id}/work`
- 生成内容和编译产物全部落在 `tmp/` 和 `outputs/`

## 8. 审查结论

本地 MVP 的正确方向是：

用户输入内容
→ 系统抽取结构
→ 映射到 `latex-scnu` 工作模板
→ 生成 `.tex` 工程
→ 本地编译
→ 导出 PDF 和 zip

只要这条链路跑通并且错误可诊断，本轮就达成目标。
